## CRITICAL Findings

### C1. Arbitrary Code Execution via Validation Scripts

**File:** `scripts/worker.sh:235`
**Status:** ACCEPTED RISK (documented)

Task markdown files contain `## Validation` blocks extracted and executed verbatim:

```bash
VALIDATION_OUTPUT=$(timeout 120s bash -c "$validation_script" 2>&1)
```

AI-generated tasks (from `generate-tasks.sh` or manager fix-tasks) write arbitrary bash into this field. A compromised or hallucinating AI model produces code that runs with full user privileges.

**Mitigation:** This is inherent to the system design. Users are warned in README and SECURITY.md. Validation scripts run with `timeout` to limit duration. Future work: sandbox via containers.

---

### C2. Claude CLI Runs with --dangerously-skip-permissions

**File:** `scripts/worker.sh:284`
**Status:** ACCEPTED RISK (documented)

```bash
timeout ... setsid $CLAUDE_BIN --dangerously-skip-permissions -p "$task_content"
```

Claude runs with zero sandboxing — it can read/write any file, execute any command, and make network requests on the worker machine.

**Mitigation:** This is a fundamental design requirement for autonomous code generation. The blast radius is now limited by C7 fix (explicit `git add` staging prevents arbitrary file propagation). Users are warned in README.

### C4. SSH Command Injection from Config

**File:** `scripts/start-night.sh:218-221`
**Status:** ACCEPTED RISK (documented)

```bash
ssh "$ssh_alias" "cd ~/project && $cmd"
```

`setup_commands` from `config.json` are executed verbatim on remote machines via SSH.

**Mitigation:** `config.json` is excluded from git via `.gitignore` and is operator-controlled. This is a configuration-level trust boundary — the operator who writes `config.json` has intentional access to all machines. Documented in trust model below.

### C5. Telegram Bot `execSync` Injection — FIXED

**File:** `scripts/telegram-bot.js:204,218`
**Status:** FIXED

Added input validation functions:

- `isValidSshAlias()` — validates SSH alias is alphanumeric/dots/hyphens
- `isValidRole()` — validates role is alphanumeric/hyphens
- `sanitizeErrorMessage()` — strips file paths from error messages sent to Telegram

All `execSync` calls now validate inputs before interpolation.

### C7. `git add -A` Stages All AI-Created Files — FIXED

**File:** `scripts/worker.sh`, `scripts/manager-loop.sh`, `scripts/start-night.sh`
**Status:** FIXED

All instances of `git add -A` replaced with explicit staging:
- **worker.sh:** `git add "$role_dir/" tasks/ results/` — only stages the role's source directory, task state changes, and result files
- **manager-loop.sh:** `git add tasks/ results/ .ralph/skills/` — only stages review artifacts
- **start-night.sh:** `git add tasks/` or `git add tasks/ results/ .ralph/ src/ scripts/`

This prevents AI-created files outside expected directories from propagating to other agents.

## Architectural / Systemic Risks

These are inherent to the design and documented as accepted risks:

### 1. AI-to-Shell Execution Pipeline

AI output flows into `bash -c`, `git commit`, file paths, and SSH commands. The `--dangerously-skip-permissions` flag explicitly removes safety boundaries. This is the core architectural trade-off enabling autonomous AI coding.

**Mitigation applied:** Explicit `git add` staging (C7 fix) limits file propagation. `$ROLE` validation (H1 fix) prevents path traversal.

### 2. `config.json` is a Single Point of Compromise

Nearly all scripts source roles, SSH aliases, setup commands, and directory paths from it.

**Mitigation:** `config.json` is in `.gitignore` and is operator-controlled locally. It never enters the git repository.

### 3. Git as Shared Trust Medium

All agents share a repository. AI-created files could propagate to every agent.

**Mitigation applied:** `git add -A` replaced with explicit staging (C7 fix). Only files in expected directories (`src/$ROLE/`, `tasks/`, `results/`) are committed.

### 4. No Code Review Gate

Generated code is committed and pushed automatically. There is no human-in-the-loop before code reaches other machines.

**Mitigation:** The manager's review loop (`manager-loop.sh`) provides automated AI review with escalation to `tasks/review/` for human attention. Morning review script (`morning-review.sh`) facilitates human oversight.

## Trust Model

Ralph Loops operates with the following trust boundaries:

| Component | Trust Level | Rationale |
|-----------|------------|-----------|
| Operator (you) | Full trust | Writes `config.json`, starts system, reviews results |
| `config.json` | Operator-controlled | Never committed to git; defines SSH, roles, commands |
| AI models (Claude/Gemini) | Untrusted output | AI output treated as potentially hostile; validated where possible |
| Task files | Semi-trusted | Generated by AI or operator; validation blocks execute in shell |
| Git repository | Shared medium | Now restricted to explicit file staging |
| Worker machines | Trusted execution | SSH key access controlled by operator |

**Users must understand:** This system grants AI unrestricted code execution on worker machines. It is designed for trusted, isolated networks (Tailscale VPN) where the operator accepts this risk in exchange for autonomous productivity.
